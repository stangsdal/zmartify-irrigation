# 12 Alarm Manager

---

# 12.1 Purpose

The Alarm Manager is responsible for detecting, prioritizing, recording, distributing and managing all alarms generated by the Zmartify Irrigation Controller.

It serves as the central authority for alarm processing and ensures that abnormal conditions are handled consistently throughout the firmware.

The Alarm Manager shall:

* Protect the irrigation system
* Protect connected equipment
* Inform the user
* Inform the smart home
* Record historical events
* Support diagnostics
* Support predictive maintenance

The Alarm Manager shall never communicate directly with hardware.

---

# 12.2 Design Objectives

The Alarm Manager shall:

* Receive alarms from every subsystem
* Prioritize alarms
* Prevent duplicate alarms
* Support alarm acknowledgement
* Support alarm clearing
* Record alarm history
* Publish alarms via MQTT
* Display alarms on the local HMI
* Trigger emergency shutdown when required

---

The Alarm Manager shall receive alarms exclusively through the Event Bus.

# 12.4 Alarm Philosophy

Every alarm follows the same lifecycle.

```text
Condition Detected

↓

Alarm Created

↓

Classification

↓

Notification

↓

Logging

↓

User Acknowledgement

↓

Condition Removed

↓

Alarm Cleared
```

This ensures deterministic alarm behaviour throughout the controller.

# 12.6 Alarm Severity

Four alarm levels are defined.

## Information

Normal operational events.

Examples:

* Firmware updated
* Configuration saved
* Wi-Fi connected

No user action required.

---

## Warning

System continues operating.

Examples:

* Weak Wi-Fi
* Weather unavailable
* Cabinet door open
* High cabinet temperature

---

## Critical

Immediate operator attention required.

Examples:

* Pressure collapse
* Major leak
* Flow mismatch
* Master valve fault

Critical alarms may stop irrigation.

---

## Emergency

Immediate protective shutdown.

Examples:

* Emergency Stop
* Multiple simultaneous failures
* Internal firmware safety violation

Emergency alarms always initiate Safe Shutdown.

---

# 12.7 Alarm Priorities

| Priority    | Response           |
| ----------- | ------------------ |
| Emergency   | Immediate Shutdown |
| Critical    | Stop Irrigation    |
| Warning     | Continue Operation |
| Information | Log Only           |

Priority determines processing order within the Event Bus.

# 12.11 Alarm Suppression

To prevent alarm flooding:

Duplicate alarms shall be suppressed.

Example:

```text
Pressure Low

↓

Already Active?

↓

Yes

↓

Increase Counter

↓

Do NOT create new alarm
```

Each alarm shall maintain an occurrence counter.

---

# 12.12 Alarm Correlation

Related alarms shall be correlated.

Example:

```text
Pressure Collapse

+

Flow High

↓

Pipe Burst
```

Instead of presenting multiple independent alarms, the controller shall generate a higher-level diagnosis when confidence is sufficient.

Future firmware may use rule-based or AI-assisted correlation.

---

# 12.13 Safe Shutdown Integration

Critical alarms may request Safe Shutdown.

Sequence:

```text
Alarm Raised

↓

Safety Supervisor

↓

Cancel Irrigation

↓

Close Zones

↓

Close Master Valve

↓

Disable Relays

↓

Publish Alarm

↓

Fault State
```

The Alarm Manager requests shutdown; the Safety Supervisor executes it.

# 12.25 Engineering Notes

The Alarm Manager is intended to be much more than a conventional alarm list.

By combining event correlation, suppression of duplicate alarms and intelligent prioritization, it becomes a decision-support system that helps users identify the actual root cause rather than presenting a flood of unrelated events.

This architecture also provides an excellent foundation for future machine-learning-based diagnostics.

---

# 12.26 Chapter Summary

The Alarm Manager is the central event handling and fault management subsystem of the Zmartify Irrigation Controller.

It provides consistent processing of all abnormal conditions, coordinates safe system responses and maintains a complete historical record of system behaviour.

By integrating tightly with the Event Bus, Flow Manager, Pressure Manager and Safety Supervisor, it enables the controller to react quickly to faults while presenting users with clear, actionable information. This architecture significantly enhances reliability, serviceability and long-term maintainability.
